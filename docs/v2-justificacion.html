Decisión de Arquitectura

Por qué se migró a Iris Bot v2

Limitaciones de la primera versión y cómo cada decisión de diseño en v2 responde a un problema concreto.

Contexto

La primera versión (TelegramBotIA_GS) fue construida como prueba de concepto: un asistente que recibía un mensaje, clasificaba su intención y generaba una respuesta. Funcionó bien para consultas simples, pero mostró sus límites cuando los requerimientos crecieron: más herramientas, más usuarios, distintos roles, trazabilidad de costos y comportamientos diferenciados por área.

Limitaciones de v1 y su solución en v2

Cada mejora responde directamente a una fricción encontrada en producción.

1

El agente solo sabía hacer una cosa por mensaje

Problema v1

El flujo era lineal — clasifica → genera SQL o busca en Knowledge → responde. Si la respuesta requería combinar dos fuentes o calcular sobre un resultado, era imposible en un solo mensaje.

Solución v2

Se adoptó el paradigma ReAct (Reasoning + Acting). El agente razona paso a paso y puede invocar múltiples herramientas en secuencia: consultar BD, calcular un porcentaje y formatear, todo en un mensaje.

2

Agregar una herramienta nueva requería modificar código

Problema v1

Las herramientas (/ia, /alertas) eran handlers de Telegram hardcoded. Añadir una nueva: editar varios archivos, hacer deploy, reiniciar el bot.

Solución v2

Las herramientas son clases independientes registradas en ToolRegistry. El agente las descubre desde el catálogo BotIAv2_Recurso en BD. Agregar una tool: crear la clase, registrarla en factory.py, insertar una fila.

3

El comportamiento del agente era hardcoded

Problema v1

El system prompt, temperatura e instrucciones vivían en plantillas Jinja2 en el código. Ajustar el tono del bot requería un pull request y un redeploy.

Solución v2

Cada agente está definido en BotIAv2_AgenteDef. El system prompt, temperatura y max iteraciones se editan directamente en BD. Los cambios toman efecto en el siguiente mensaje. El historial de versiones se guarda vía trigger automático.

4

No había memoria entre sesiones

Problema v1

Cada mensaje era independiente. El bot no recordaba que un usuario había preguntado lo mismo ayer, ni sus preferencias, ni su contexto laboral.

Solución v2

3 capas de memoria en UserMemoryProfiles:

  • Contexto laboral del usuario
  • Temas recientes consultados
  • Historial breve de interacciones

Más preferencias explícitas via save_preference: alias, idioma, formato.

5

Los permisos eran todo o nada por comando

Problema v1

Un usuario tenía permiso para /alertas o no lo tenía. No era posible diferencias como "solo Gerencia de Infraestructura puede ver alertas críticas" sin código adicional.

Solución v2 — SEC-01
usuario → override definitivo rol → permisivo gerencia → permisivo dirección→ permisivo sin regla → denegado por defecto

Cada permiso tiene fecha de expiración. Todo administrable desde BD.

6

No había visibilidad de costos ni de lo que hacía el agente

Problema v1

Se sabía si una operación fue exitosa o falló, pero no cuántos tokens consumió, cuánto costó en USD, cuántos pasos dio el agente ni qué herramienta usó.

Solución v2
TablaQué guarda

InteractionStepsCada paso del loop ReAct: thought, tool, tokens

7

Un solo agente para todos los casos de uso

Problema v1

El mismo agente respondía datos, RRHH, alertas y conversación casual con el mismo prompt. Optimizar un flujo afectaba los demás.

Solución v2

IntentClassifier elige el agente según la intención:

  • Datos — SQL y análisis numérico
  • Conocimiento — base institucional
  • Casual — conversacional sin tools
  • Generalista — fallback automático

Cada agente tiene su propio prompt, temperatura y tools asignadas.

8

Todo el acceso a BD era SQL directo en el código

Problema v1

Los queries de auth, permisos y operaciones vivían como strings SQL en archivos Python. Cambiar la estructura de una tabla implicaba buscar en todo el código.

Solución v2

28 stored procedures con prefijo homologado BotIAv2_sp_*. El código solo llama:

EXEC BotIAv2_sp_GetUsuarioByChatId @telegramChatId = :chat_id

Si cambia la estructura interna, solo se actualiza el SP. La aplicación no se toca.
